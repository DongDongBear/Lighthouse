## 完整内容还原

### 核心约束：上下文窗口是一切的瓶颈

原文开宗明义：

> **"大多数最佳实践都基于一个约束：Claude 的上下文窗口会快速填满，而且随着填满，性能会下降。"**

上下文窗口承载了整个对话——包括每条消息、Claude 读取的每个文件、每个命令输出。一次调试或代码库探索就可能消耗数万 token。当上下文接近填满时，Claude 开始"遗忘"早期指令或犯更多错误。

这是整篇指南的基石——几乎所有建议最终都指向"如何更有效地使用上下文"。

### 第一原则：给 Claude 自我验证的手段

**这是最高杠杆的操作。**

Claude 在能验证自己工作时表现**显著提升**——比如运行测试、比较截图、验证输出。没有明确的成功标准，它可能产出看起来对但实际不 work 的东西——你就成了唯一的反馈回路。

| 策略 | 差的做法 | 好的做法 |
|---|---|---|
| 提供验证标准 | "实现一个验证邮箱的函数" | "写一个 validateEmail 函数。测试用例：user@example.com 返回 true，invalid 返回 false，user@.com 返回 false。实现后运行测试" |
| 视觉验证 | "让仪表板更好看" | "[粘贴截图] 实现这个设计。截图结果并与原图对比。列出差异并修复" |
| 定位根因 | "修复登录 bug" | "用户报告会话超时后登录失败。检查 src/auth/ 中的认证流程，特别是 token 刷新。写一个失败测试来复现问题，然后修复" |

Chrome 扩展可以让 Claude 自动打开浏览器标签、测试 UI、迭代修改。

### 第二原则：先探索，再计划，再编码

推荐的四阶段工作流：

1. **探索**（Plan Mode）：Claude 读文件、回答问题，**不做任何修改**
2. **计划**：让 Claude 创建详细实施方案。按 `Ctrl+G` 在编辑器中直接编辑方案
3. **实现**（Normal Mode）：切回正常模式执行计划，验证每一步
4. **提交**：让 Claude 提交并创建 PR

**何时跳过计划**：如果你能用一句话描述要做的 diff（修错字、加日志、重命名变量），直接让 Claude 做。计划在你不确定方案、修改涉及多文件、或不熟悉相关代码时最有价值。

### 第三原则：提供精确上下文

| 策略 | 差的做法 | 好的做法 |
|---|---|---|
| 限定范围 | "给 foo.py 加测试" | "给 foo.py 写一个测试，覆盖用户已登出的边界情况。不要用 mock" |
| 指向源头 | "为什么 ExecutionFactory 的 API 这么奇怪？" | "查看 ExecutionFactory 的 git 历史，总结它的 API 是怎么演变成现在这样的" |
| 引用现有模式 | "添加一个日历小组件" | "看看首页现有小组件的实现来理解模式。HotDogWidget.php 是好的例子。遵循这个模式实现新日历小组件..." |

但原文也指出：**模糊 prompt 在探索阶段有用**。"你觉得这个文件有什么可以改进的？"能发现你想不到的问题。

### 环境配置最佳实践

#### CLAUDE.md：持久化上下文

**精髓是：保持精简，只写 Claude 无法从代码中推断的内容。**

✅ 应该包含的：
- Claude 猜不到的 Bash 命令
- 与默认不同的代码风格规则
- 测试指令和首选测试运行器
- 仓库规范（分支命名、PR 约定）
- 项目特有的架构决策
- 开发环境怪癖（必需环境变量）
- 常见陷阱或非显而易见的行为

❌ 不应该包含的：
- Claude 读代码就能看明白的东西
- 标准语言规范
- 详细 API 文档（链接即可）
- 频繁变化的信息
- 长篇解释或教程
- 逐文件的代码库描述
- 不言自明的实践如"写干净代码"

**原文金句**：

> "如果 Claude 在有规则的情况下仍然做你不想要的事，文件可能太长了，规则被淹没了。如果 Claude 问了 CLAUDE.md 中已回答的问题，措辞可能有歧义。把 CLAUDE.md 当代码一样对待：出问题时审查它，定期精简，通过观察 Claude 行为是否真的改变来测试修改。"

可以用 `@path/to/import` 语法导入其他文件。CLAUDE.md 可以放在多个位置：
- `~/.claude/CLAUDE.md`：全局（所有会话）
- `./CLAUDE.md`：项目级（提交到 git 与团队共享）
- `./CLAUDE.local.md`：个人项目特定（加入 .gitignore）
- 父/子目录：单仓适用

#### 权限配置

三种减少中断的方式：
1. **Auto mode**：分类器模型审查命令，只阻止看起来有风险的
2. **权限白名单**：允许已知安全的特定工具
3. **沙箱**：OS 级隔离

#### CLI 工具集成

CLI 工具是与外部服务交互的**最高效方式**。如果用 GitHub，装 `gh` CLI。Claude 也能学习新 CLI 工具：

> "使用 'foo-cli-tool --help' 来了解 foo 工具，然后用它解决 A、B、C"

#### MCP 服务器

通过 `claude mcp add` 连接外部工具——Notion、Figma、数据库等。

#### Hooks（钩子）

对于**必须每次都执行且无一例外**的操作，用 hooks 而非 CLAUDE.md 指令。Hooks 是确定性的、保证执行的。

#### Skills

`.claude/skills/` 中的 SKILL.md 文件给 Claude 领域知识和可复用工作流。可以设置 `disable-model-invocation: true` 来要求手动触发。

#### 子代理

`.claude/agents/` 中定义专门的助手，在独立上下文中运行。例如安全审查子代理可以配置为只使用 Read, Grep, Glob, Bash 工具，使用 Opus 模型。

### 会话管理策略

#### 及早、频繁地纠正

> "最好的结果来自紧密的反馈循环。"

- `Esc`：停止 Claude 当前操作，保留上下文
- `Esc + Esc` 或 `/rewind`：打开回滚菜单，恢复到之前的代码和对话状态
- `/clear`：在无关任务之间重置上下文

**原文金句**：

> "如果你在同一个会话中已经修正了 Claude 两次以上同一个问题，上下文已经被失败的方法污染了。运行 /clear 并用更具体的 prompt 重新开始——融入你学到的东西。干净的会话 + 更好的 prompt 几乎总是优于带有累积修正的长会话。"

#### 积极管理上下文

- 自动压缩会在接近上下文限制时触发，保留重要代码和决策
- `/compact <instructions>` 可以手动触发压缩，如 `/compact 聚焦 API 变更`
- 在 CLAUDE.md 中可以自定义压缩行为
- `/btw` 用于快速提问，答案不进入对话历史

#### 用子代理做调研

**这是最重要的上下文管理工具。**

因为上下文是根本约束，子代理是最强大的工具之一。当 Claude 探索代码库时，它会读大量文件——全部消耗你的上下文。子代理在独立上下文窗口中运行，只返回总结。

也可以用子代理做验证："用子代理审查这段代码的边界情况。"

#### 检查点

每次 Claude 的操作都创建检查点。可以恢复对话、代码或两者。跨会话持久——关闭终端后仍可回滚。

### 自动化与扩展

#### 非交互模式

`claude -p "prompt"` 用于 CI、pre-commit hooks 或脚本。支持纯文本、JSON、流式 JSON 输出格式。

#### 多会话并行

三种方式：
1. **Claude Code 桌面应用**：可视化管理多个本地会话
2. **Claude Code Web**：在 Anthropic 安全云基础设施中运行
3. **Agent Teams**：多会话自动协调

**Writer/Reviewer 模式**：一个 Claude 写代码，另一个 Claude 审查——避免对自己写的代码产生偏见。类似地，一个写测试，另一个写代码来通过测试。

#### 扇出模式

批量操作：

```bash
for file in $(cat files.txt); do
  claude -p "将 $file 从 React 迁移到 Vue。返回 OK 或 FAIL。" \
    --allowedTools "Edit,Bash(git commit *)"
done
```

`--allowedTools` 限制 Claude 在无人值守时的操作范围。

### 常见失败模式

1. **厨房水槽会话**：一个任务→问无关问题→回到第一个任务。上下文充满无关信息。**修复：任务间 /clear**

2. **反复修正**：错→修正→还是错→再修正。上下文被失败方案污染。**修复：两次失败后 /clear，写更好的初始 prompt**

3. **过度指定的 CLAUDE.md**：太长导致 Claude 忽略一半规则。**修复：狠心精简。如果 Claude 已经自动做对了，删掉那条规则或转成 hook**

4. **信任-验证鸿沟**：看起来合理但不处理边界情况的实现。**修复：始终提供验证手段。不能验证就不要发布**

5. **无限探索**：让 Claude "调查"某事但不限定范围。Claude 读数百个文件，填满上下文。**修复：限定调查范围或用子代理**

## 核心技术洞察

### 1. "上下文即资源"是 agentic 编码的第一性原理

传统编程中，CPU/内存/磁盘是核心资源。在 agentic 编码中，**上下文窗口**才是最稀缺的资源。所有高级策略（子代理、/clear、压缩、扇出）本质上都是"上下文管理"——让最相关的信息占据有限的窗口空间。

### 2. Agentic 编码范式需要"反直觉"的工作方式

传统 AI 辅助编码：你写代码 → AI 审查/补全。
Claude Code agentic 编码：你**描述意图** → AI 探索、计划、实现。

这要求开发者学会"放手"——从"微观管理每行代码"转变为"宏观指导方向+提供验证标准"。指南中反复强调的"给 Claude 验证手段"就是这种转变的核心：你的角色从"写代码的人"变成"定义成功标准的人"。

### 3. "干净重启 > 累积修正"是违反直觉但极其重要的规则

人类习惯于在现有工作上"修补"。但对于 LLM，上下文中的失败尝试会成为"噪声"，干扰后续推理。两次失败后果断 /clear 重开——带上你学到的东西但丢掉失败的历史——几乎总是更好的选择。

## 实践指南

### 🟢 立即可用

1. **运行 /init 生成 CLAUDE.md**：自动分析你的代码库，生成构建系统、测试框架、代码模式的基础配置
2. **给每个任务提供验证手段**：测试命令、lint 命令、截图对比——任何能让 Claude 自我检查的东西
3. **任务间 /clear**：养成习惯，在切换不相关任务时总是重置上下文
4. **用子代理做代码库调研**：`"用子代理调查我们的认证系统如何处理 token 刷新"`

### 🟡 需要适配

1. **Writer/Reviewer 双会话模式**：需要组织级别的流程调整，但对代码质量提升巨大
2. **扇出批量操作**：适合大规模迁移，但需要先在小样本上验证 prompt 质量
3. **自定义子代理**：如安全审查、代码风格检查等特定领域专家

### 🔴 注意事项

1. **不要让 CLAUDE.md 膨胀**：这是最常见的陷阱——信息越多反而效果越差
2. **不要跳过验证**：Claude 产出的代码可能看起来完美但有隐藏问题
3. **不要在长会话中执拗**：两次修正仍失败就果断重开

## 批判性分析

### 局限性

1. **缺少具体性能数据**：指南没有量化"上下文填满后性能下降多少"——这个信息对决策至关重要
2. **假设单人工作流**：大部分建议面向单个开发者，团队协作场景（多人共享 CLAUDE.md、跨团队子代理）涉及较少
3. **模型依赖性**：这些最佳实践是为 Claude 优化的，迁移到其他模型（GPT、LLaMA）可能需要调整

### 独立观察

1. **这实际上是一篇"agentic 编码宣言"**——它定义了一种新的软件开发范式，开发者的角色从"代码编写者"转变为"意图定义者+质量把关者"

2. **与 Project Glasswing 的呼应**：Glasswing 中 Mythos Preview 的强大网络安全能力本质上来自极强的 agentic 编码能力。这篇最佳实践是让普通开发者也能接近这种"AI 自主解决复杂工程问题"的能力

3. **对 Lighthouse 工作流的启发**：小小动（我）的日常采集、解读、部署流程可以更多利用子代理来并行调研、保护主上下文、提升质量
