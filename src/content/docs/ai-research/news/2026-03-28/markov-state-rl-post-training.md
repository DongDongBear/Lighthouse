## 核心 Insight

RL 后训练（RLHF、GRPO 等）已经成为 LLM 能力提升的标准流程。但一个令人不安的经验现象是：**RL 后训练更像是"精炼"而非"发现"——它能让模型更好地使用已有能力，却难以帮助模型获得全新的问题解决策略。**

这篇论文从理论上解释了这一现象的根源：**结构性瓶颈在于状态表示，而非算法本身。**

当前 LLM 的 RL 后训练中，模型的"状态"是整个动作历史序列（action history）——即从任务开始到当前为止的所有 token。随着推理步骤增多，这个"状态"指数级膨胀。具体来说：

- **动作序列表示：** 状态空间大小为 |A|^H（动作空间大小的步骤数次方），指数级增长
- **Markov 状态表示：** 状态空间大小与 H 多项式相关，增长可控

这就像两种不同的解谜策略：
- **动作序列策略：** 记住"第一步向左，第二步向上，第三步向右……"——步骤越多，需要记住和搜索的路径就越多，呈指数级爆炸
- **Markov 状态策略：** 只看当前棋盘局面，根据局面选择下一步——不管之前走了多少步，决策空间始终一样大

### Combination Lock 证明

论文用一个精巧的"组合锁"问题来量化这个差距：

- 一个 H 步的序列决策问题，每步有 |A| 个选项
- 只有一条正确路径能解锁
- **Markov 代理（能观察当前锁的状态）：** 30K 步内收敛
- **动作序列代理（只能记住历史动作）：** 800K 步后仍未收敛

30K vs 800K+，差距超过 26 倍——而且随着问题规模增大，这个差距会进一步指数级扩大。

### 为什么现有方法没有发现这个问题？

因为现有的 RL 后训练评估主要集中在数学推理和代码生成这类任务上——这些任务的一个共同特点是，动作历史本身就包含了大部分有用的状态信息（比如，数学推导的中间步骤已经编码了当前的推理进度）。只有在 Sokoban（推箱子）、Futoshiki（不等式数独）这类需要"空间状态跟踪"的任务上，动作序列表示和 Markov 状态表示之间的差距才会充分暴露。

## 方法详解

### 理论框架：两个关键命题

#### Proposition 1: 动作序列表示的指数下界

在基于动作序列的 RL 后训练中，策略 pi 需要从完整的动作历史 (a_1, a_2, ..., a_t) 映射到下一个动作。对于 Combination Lock 问题：

- 有效策略空间大小为 |A|^H
- 找到最优策略所需的样本复杂度下界为 Omega(|A|^H)
- 这意味着随着步骤数 H 增长，所需样本量**指数级**增长

#### Proposition 2: Markov 状态表示的多项式上界

当代理能观察到当前环境的 Markov 状态 s_t 时：

- 有效策略空间大小与 |S| * |A| * H 成多项式关系
- 找到最优策略所需的样本复杂度上界为 O(poly(|S|, |A|, H))
- 这意味着随着步骤数 H 增长，所需样本量仅**多项式级**增长

两个命题的对比直接解释了"能力天花板"的来源：不是 RL 算法不够强，而是动作序列表示本身就注定了指数级的搜索难度。

### 实验方法：如何为 LLM 引入 Markov 状态

论文在三个代表性任务上实现了 Markov 状态的引入：

#### Sudoku（数独）

- **动作序列表示：** 模型输出一系列填数动作，历史是所有已填数字的序列
- **Markov 状态表示：** 在每步决策前，将当前的完整棋盘（9x9 网格，含已填和未填格子）作为状态呈现给模型
- **状态更新：** 每次填数后，更新棋盘状态

#### Sokoban（推箱子）

- **动作序列表示：** 模型输出上/下/左/右动作序列
- **Markov 状态表示：** 在每步前，渲染当前地图（玩家位置、箱子位置、目标位置）
- **状态更新：** 每次移动后，重新渲染整个地图

#### Futoshiki（不等式数独）

- **动作序列表示：** 模型输出一系列填数动作
- **Markov 状态表示：** 在每步前，展示当前网格和所有不等式约束的满足情况
- **状态更新：** 每次填数后，更新网格并标注约束状态

**关键设计：** 在所有任务中，Markov 状态的引入不需要修改模型架构或 RL 算法——仅需修改提示模板，在每步推理前注入当前状态的文本描述。这意味着该方法可以直接应用于任何现有的 LLM RL 后训练流程。

### 训练细节

- **基础模型：** Qwen3-4B
- **RL 算法：** GRPO（Group Relative Policy Optimization）
- **奖励设计：** 正确完成任务为 +1，失败为 0
- **对比条件：** 相同的模型、算法、奖励，唯一区别是状态表示方式

## 实验结果

### 主实验：Markov 状态 vs 动作序列

基础模型 Qwen3-4B 在三个任务上的表现：

| 任务 | Action-seq + RL | Markov + RL | 提升幅度 |
|---|---|---|---|
| Sudoku | 93.5% | 97.1% | +3.6pp |
| Sokoban | 2.3% | 76.1% | +73.8pp |
| Futoshiki | 0.1% | 75.0% | +74.9pp |

**关键观察：**

1. **Sudoku 差距小：** 数独任务中，动作序列本身已经编码了大部分有用的状态信息（每一步填数隐含了当前棋盘状态），因此 Markov 状态的额外收益有限。这解释了为什么现有 RL 后训练在数学推理任务上效果不错。

2. **Sokoban 和 Futoshiki 差距巨大：** 这两个任务需要追踪空间状态——推箱子需要知道"当前箱子和玩家在哪里"，不等式数独需要知道"当前网格和约束满足情况"。动作序列无法高效编码这些信息，导致 RL 几乎完全失败。

3. **2.3% 和 0.1% 的含义：** 在 Sokoban 和 Futoshiki 上，动作序列 + RL 的表现与随机猜测几乎无异。这不是"效果差一点"——这是"完全无法学习"。

### OOD 泛化实验

在分布外（Out-of-Distribution）测试集上的表现——使用更大或更复杂的问题实例：

| 任务 | Action-seq + RL (OOD) | Markov + RL (OOD) |
|---|---|---|
| Sokoban | 0.0% | 31.6% |
| Futoshiki | 0.0% | 42.6% |

这组数据更加触目惊心：

- **动作序列方法完全无法泛化：** 0.0% 的 OOD 准确率意味着模型连最基本的迁移能力都没有习得
- **Markov 状态方法展现了真正的泛化：** 31.6% 和 42.6% 虽然不算高，但说明模型学到了可迁移的问题解决策略，而不仅仅是在训练分布上的记忆

### Combination Lock 理论验证

| 代理类型 | 收敛所需步数 | 最终成功率 |
|---|---|---|
| Markov 代理 | ~30,000 | 接近 100% |
| Action-seq 代理 | >800,000 | 未收敛 |

该实验直接验证了 Proposition 1 和 Proposition 2 的预测：指数 vs 多项式的差距在实践中体现为数量级的效率差异。

## 批判性分析

### 局限性

1. **任务选择偏差：** 论文选择的三个任务（数独、推箱子、不等式数独）都是完全可观测的确定性环境——Markov 状态是精确可定义的。但对于大量现实任务（开放域对话、创意写作、复杂推理），什么构成"Markov 状态"本身就是一个未解问题。

2. **状态表示的人工设计：** 实验中的 Markov 状态是由人工定义的（棋盘布局、地图渲染等）。论文没有解决"如何自动从任务中提取 Markov 状态"这一关键问题。如果需要人工为每个任务设计状态表示，方法的可扩展性受限。

3. **模型规模局限：** 所有实验仅在 Qwen3-4B 上进行。大模型（70B+）可能通过更强的内部状态追踪能力部分弥补动作序列表示的不足——换句话说，"能力天花板"可能随模型规模变化。

4. **与 CoT 的关系：** Chain-of-Thought 推理本质上是一种在动作序列中编码中间状态的策略。论文没有深入讨论 CoT 在多大程度上已经缓解了动作序列表示的问题。

### 实际应用路径

论文提出了三个具有高实用价值的应用方向：

1. **编程任务：** 状态 = 当前代码快照。每次代码修改后，将完整的当前代码（而非修改历史）作为状态提供给模型。这与目前 Cursor/Copilot 等 AI 编程工具的工作方式高度契合。

2. **数学推理：** 状态 = 已确立的引理和中间结论。不需要回顾整个推导过程，只需要知道"目前已经证明了什么"。这对长链条数学推理特别有价值。

3. **迭代精炼：** 状态 = 当前草稿版本。写作、设计等迭代任务中，模型只需要看到当前版本而非完整的修改历史。

### 独立观察

- **与同期工作的关联：** 这篇论文与同日 Meta 团队发表的 LLM-as-Judge 对抗性研究形成有趣的呼应——两篇论文都指向一个共同主题：**当前 RL 后训练的成功可能部分建立在评估任务的特殊性上，而非方法本身的通用有效性。** Markov 状态论文在"能力获取"维度揭示了这一点，LLM-as-Judge 论文在"奖励信号"维度揭示了类似的脆弱性。

- **对 Agent 架构的启示：** 当前主流 Agent 框架（ReAct、Reflexion 等）已经在某种程度上使用了"状态注入"——在每步推理前注入环境观察。这篇论文为这一实践提供了坚实的理论基础：Agent 架构的有效性可能不仅来自"工具使用"，更来自"状态表示的改善"。

- **RL 后训练方法论的重新审视：** 如果能力天花板确实来自状态表示而非算法，那么 RL 后训练研究的重点可能需要从"更好的 RL 算法"转向"更好的状态表示设计"。这是一个范式层面的方向调整。

### 对领域的影响

短期：为 RL 后训练在推理任务（特别是需要状态追踪的任务）上提供了即时可用的改进方案——修改提示模板注入状态即可。

中期：推动 Agent 框架和 RL 后训练的融合——Agent 的"环境观察"机制本质上就是 Markov 状态注入，两个领域可能走向统一。

长期：激发"自动状态抽象"的研究——让模型自己学习如何从动作历史中提取紧凑的 Markov 状态，这可能是突破 RL 后训练能力天花板的终极方案。
