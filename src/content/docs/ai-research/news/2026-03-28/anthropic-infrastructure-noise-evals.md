## 文章背景

Agent 编程评测（Agentic Coding Evals）已成为衡量大模型"真实编程能力"的核心战场。SWE-bench、Terminal-Bench 等基准被广泛引用，排行榜上 1-2 个百分点的差距经常被解读为"模型 A 比模型 B 更强"。

但这里存在一个被系统性忽视的问题：**这些评测运行在容器化环境中，容器的资源配置（CPU、内存的分配上限和下限）本身就是一个实验变量。** 当排行榜上的不同参赛方使用不同的基础设施配置时，我们无法区分得分差异来自模型能力还是来自基础设施条件。

Anthropic 工程团队的这篇文章首次系统性地量化了这个问题。他们的核心发现直白而令人不安：仅通过调整容器资源配置，同一个模型在同一个基准上的得分差异可达 6 个百分点——这个数字在多数排行榜上足以改变模型排名。

## 完整内容还原

### 根本原因：容器运行时的"保证分配"与"硬杀阈值"

文章首先解释了问题的技术根源。容器运行时（如 Kubernetes/GKE 中的 containerd）通过两个参数控制资源：

- **Guaranteed allocation（保证分配 / 地板值）：** 容器至少能获得的资源量
- **Hard kill threshold（硬杀阈值 / 天花板值）：** 容器最多能使用的资源量，超过即被 OOM kill

当这两个值被设为相同数值时，容器的资源余量为零。任何瞬时的资源尖峰——哪怕是操作系统级别的正常波动——都会触发 OOM kill，导致 Agent 任务直接失败。

这个机制在 chatbot 场景下影响不大（单次请求资源消耗可预测），但在 Agent 编程场景下影响巨大：Agent 需要安装依赖包、编译代码、启动子进程、运行测试，这些操作的资源消耗高度不确定且存在尖峰。

### 实验设计

Anthropic 在 Google Kubernetes Engine (GKE) 上设计了严格的控制实验：

**6 种资源配置：**

| 配置 | 描述 |
|---|---|
| 1x（严格） | 天花板 = 地板，零余量 |
| 3x | 天花板 = 3 倍地板 |
| 5x | 天花板 = 5 倍地板 |
| 7x | 天花板 = 7 倍地板 |
| 10x | 天花板 = 10 倍地板 |
| Uncapped（无上限） | 仅设地板，不设天花板 |

**控制变量：** 同一个 Claude 模型、同一个评测框架（harness）、同一个任务集。唯一变化的是容器资源配置。

这是一个教科书级别的控制实验设计——所有自变量被锁死，只暴露一个实验变量。

### 核心结果

#### 基础设施错误率的梯度变化

| 配置 | 基础设施错误率 |
|---|---|
| 1x（严格） | 5.8% |
| 3x | 2.1% |
| Uncapped | 0.5% |

从 1x 到 3x，基础设施错误率从 5.8% 大幅下降到 2.1%，统计显著性极高（p < 0.001）。这 3.7 个百分点的错误率差异完全来自 OOM kill 等基础设施层面的崩溃，与模型能力无关。

#### 但崩溃修复不等于得分提升

一个反直觉的发现：**1x 到 3x 的得分变化在统计上不显著（p = 0.40）。** 原因是：在 1x 配置下因 OOM 崩溃的那些任务，即使给它们更多资源不崩溃，大部分也会因为模型能力不足而失败。崩溃只是提前终止了注定失败的尝试。

这意味着仅修复基础设施崩溃不会显著改变评测结果——如果所有任务的解题策略保持不变的话。

#### 资源余量改变解题策略

真正的 6 个百分点差距来自 3x 到 uncapped 之间。在这个区间内：

- 基础设施错误率仅额外下降 1.6 个百分点（从 2.1% 到 0.5%）
- 但实际成功率跳升约 4 个百分点

差额（约 2.4 个百分点）无法用"减少崩溃"解释。这部分差距来自一个更深层的原因：**充裕的资源使得 Agent 能够采用不同的解题策略。**

文章给出了一个具体案例——bn-fit-modify 任务（贝叶斯网络拟合与修改）：

- 某些模型的第一步操作是安装 pandas + networkx + scikit-learn 三个大型依赖包
- 在资源宽裕的环境下：安装成功，模型使用这些库完成任务
- 在资源紧张的环境下：安装过程本身触发 OOM，任务直接失败

关键点在于：**不同模型默认采用不同的解题策略。** 有的模型倾向于安装重型依赖库来降低编码复杂度，有的模型倾向于用纯标准库手写解决方案。资源配置决定了哪种策略能够成功执行——这意味着基础设施配置实质上在"选择性地奖励"某些模型的编程风格。

#### 总效应

1x 到 uncapped 的总效应为 +6 个百分点（p < 0.01）。这个效应可分解为：

- 约 1.6pp：减少基础设施崩溃（但大部分崩溃任务本身也会失败）
- 约 4pp+：启用更多资源密集型的解题策略

### 交叉验证

Anthropic 不止在一个基准上验证了这个发现：

**跨模型一致性：** 方向性效应在不同模型间一致——资源更多 = 得分更高。这排除了"仅对 Claude 有效"的可能。

**SWE-bench 验证：** 在 SWE-bench 上也观察到相同模式，但效应较小——1x 与 5x 之间差距为 1.54 个百分点。效应较小可能是因为 SWE-bench 任务通常在已有代码库上修改，资源需求峰值更低。

**其他噪声源：** 文章指出基础设施噪声不限于内存配置，还包括：

- 时间限制（time limits）：过短的超时会截断合理的长时间编译或测试
- 时段性 API 延迟（time-of-day API latency）：API 调用延迟在不同时段波动
- 集群健康状态（cluster health）：节点负载、网络抖动等
- 硬件规格（hardware specs）：不同 CPU 型号的单核性能差异

这些因素叠加起来构成了一个"基础设施噪声预算"，可能远超 6 个百分点。

### 实践建议

文章给出了四条明确的操作建议：

**建议一：为每个任务同时指定地板值和天花板值。** 仅设一个参数会导致不确定行为——要么资源不足崩溃，要么无限制使用导致不同运行间资源可用性不一致。

**建议二：天花板值设为每任务规格的 3 倍作为合理折衷。** 3x 配置消除了绝大部分基础设施崩溃（错误率从 5.8% 降至 2.1%），同时不会像 uncapped 那样引入策略差异带来的不可控变量。如果目标是测量模型的"纯编程能力"，3x 是噪声与信号的最佳平衡点。

**建议三：排行榜上 3 个百分点以下的差距"值得怀疑，除非评测配置被完整记录"。** 这是文章最重要的政策建议。当基础设施配置本身就能制造 6 个百分点的差距时，任何小于这个数字一半的排名差异都可能是噪声而非信号。

**建议四：资源配置应作为"一等实验变量"。** 就像学术论文需要报告模型超参数、训练数据、随机种子一样，Agent 评测需要报告完整的基础设施配置。

文章以一句话收尾：

> "A few-point lead might signal a real capability gap -- or it might just be a bigger VM."

几分之差可能意味着真实的能力差距——也可能只是一个更大的虚拟机。

## 核心技术洞察

### 洞察一：评测基准的"测量分辨率"被基础设施噪声吞噬

物理实验中有一个基本原则：测量精度不能高于仪器精度。如果温度计的误差是 ±2°C，那么区分 36.5°C 和 37.0°C 是没有意义的。

Agent 编程评测面临完全相同的问题。当基础设施噪声的量级为 6 个百分点时，评测基准的"有效分辨率"也是 6 个百分点。在这个分辨率之下的排名差异——比如"模型 A 得 52%，模型 B 得 50%"——在统计意义上不可区分。

这不是一个理论问题。当前排行榜上的竞争格局是：

- 头部模型之间的差距通常在 1-5 个百分点
- 新模型发布时声称的"SOTA"优势通常在 2-4 个百分点
- 这些差距全部落在基础设施噪声的范围内

Anthropic 的数据直接质疑了这些声明的有效性。

### 洞察二：噪声不是随机的，而是系统性偏向特定策略

如果基础设施噪声是纯随机的（每次运行随机加减几分），问题还相对简单——多跑几次取平均值就行。但 Anthropic 的数据揭示了一个更棘手的现象：**噪声是结构化的，它系统性地有利于某些编程策略而不利于其他策略。**

具体来说：

- **资源宽裕的环境偏向"重型依赖策略"：** 安装大型库、启动多个子进程、进行资源密集型的测试——这些策略在宽裕环境下可行
- **资源紧张的环境偏向"轻量手写策略"：** 仅使用标准库、避免大型依赖、最小化子进程——这些策略在任何环境下都可行

由于不同模型天然倾向于不同策略（这与训练数据和 RLHF 偏好有关），基础设施配置实际上在**系统性地偏袒某些模型**。一个擅长"手写轻量方案"的模型在紧张环境下会获得相对优势，而一个擅长"调度重型工具链"的模型在宽裕环境下会获得相对优势。

这意味着多次运行取平均值**不能**消除这种偏差——因为它不是随机噪声，而是系统性偏差。唯一的解决方案是标准化基础设施配置。

### 洞察三：评测基准衡量的不仅是"编程能力"，还包括"环境适应力"

bn-fit-modify 案例揭示了一个深层问题：Agent 编程评测实际上在衡量一个复合能力——

1. **编程能力：** 理解问题、设计算法、编写代码
2. **环境适应力：** 根据可用资源选择合适的工具和策略

在真实软件工程中，这两种能力都很重要。一个优秀的工程师不会在 512MB 内存的容器里试图安装 TensorFlow。但在评测场景下，如果我们想衡量的是"纯编程能力"，那么"环境适应力"就是一个混淆变量。

Anthropic 的 3x 建议本质上是一种去混淆策略：给足够的资源消除大部分环境限制，使得评测结果更多地反映编程能力而非环境适应力。

## 实践指南

### 对评测基准维护者

**立即行动：**
1. 在基准文档中要求参赛方报告完整的容器资源配置（CPU 核数、内存地板值、内存天花板值、磁盘配置、网络策略）
2. 为每个任务制定推荐的资源配置规格（地板值 + 3x 天花板值）
3. 在排行榜上标注"评测配置已验证"和"评测配置未验证"两类结果

**中期改进：**
1. 引入标准化的基础设施配置规范（类似于 MLPerf 的硬件规格要求）
2. 要求参赛方在至少两种资源配置下报告结果，以量化其得分对基础设施的敏感度
3. 在排行榜上标注置信区间，而非仅显示点估计

### 对模型开发者

**评测实践：**
1. 内部评测时使用至少两种资源配置（如 3x 和 uncapped），记录差异幅度
2. 对外发布结果时附上完整的基础设施配置参数
3. 如果模型在不同资源配置下得分差异超过 2 个百分点，应在报告中标注并分析原因

**模型改进方向：**
1. 训练 Agent 具备"资源感知能力"——在执行编程任务前探测可用资源，据此选择策略
2. 评估模型是否过度依赖重型依赖策略，这种策略在资源受限的真实生产环境中也同样脆弱

### 对排行榜消费者（研究者、决策者、采购方）

1. 任何未附基础设施配置说明的排行榜结果，其排名细节不可信赖
2. 3 个百分点以内的排名差异应视为"统计上不可区分"，除非评测方提供了充分的噪声控制证据
3. 优先关注在标准化配置下取得的结果，或在多种配置下一致领先的模型

## 批判性分析

### 局限性

1. **单一云平台：** 实验在 GKE 上进行。不同云平台（AWS EKS、Azure AKS）的容器运行时实现可能引入不同的噪声特征。GKE 上的 3x 最优配置在其他平台上未必成立。

2. **模型范围有限：** 虽然文章声称"跨模型一致"，但具体测试了哪些模型、测试了多少个模型，文章并未完全披露。如果仅测试了 Anthropic 自家模型和少数几个竞争对手模型，结论的普遍性有待验证。

3. **任务集偏差：** Terminal-Bench 2.0 和 SWE-bench 的任务构成可能不代表所有 Agent 编程场景。文章中 SWE-bench 上的效应（1.54pp）明显小于 Terminal-Bench（6pp），这暗示不同类型的任务对基础设施噪声的敏感度差异很大。仅凭两个基准无法建立普适的噪声模型。

4. **未量化所有噪声源：** 文章提到了时间限制、API 延迟、集群健康、硬件规格等其他噪声源，但仅系统量化了内存配置这一个维度。总噪声预算可能远大于 6 个百分点。

5. **利益冲突考量：** Anthropic 作为排行榜的直接参与者，发表质疑排行榜可信度的研究，存在潜在的利益冲突。如果 Anthropic 的模型在当前基础设施配置下未能取得最高排名，那么质疑排行榜的有效性对 Anthropic 是有利的。这不影响研究本身的科学质量（实验设计是严谨的），但读者应注意这个背景。

### 独立观察

**与 MLPerf 的历史对比：** AI 评测领域此前经历过完全相同的问题。MLPerf 训练和推理基准在早期也面临"不同参赛方使用不同硬件配置导致结果不可比"的问题，最终通过严格的硬件规格要求和审计流程解决。Anthropic 的这篇文章实际上是在推动 Agent 评测走向 MLPerf 化——从"野生排行榜"走向"标准化基准"。

**对当前排行榜格局的影响：** 如果 3pp 以下的差距不可信，那么当前大多数"模型 A 超越模型 B"的声明都需要重新审视。这对于正在根据排行榜排名做采购决策的企业客户尤其重要——选择得分高 2 个百分点但价格贵 50% 的模型，可能是在为基础设施噪声买单。

**"更大的 VM" 效应的产业含义：** 文章揭示的不仅是评测问题。在生产环境中，Agent 编程助手同样运行在容器中。如果资源配置能影响 6 个百分点的成功率，那么企业在部署 Agent 时的基础设施投资决策（选择什么规格的容器、设置多大的资源余量）就不再是简单的成本问题——它直接影响 Agent 的有效能力。这意味着"买更好的模型"和"给更大的 VM"可能是两种等效的性能提升路径，后者往往更便宜。

**对 Agent 安全评测的连带影响：** 如果编程评测受基础设施噪声影响如此之大，那么 Agent 安全评测（如 Anthropic 自身的 RSP 评估、DeepMind 的 Frontier Safety Framework 测试）是否也面临同样的问题？安全评测通常在受控环境中运行 Agent 并观察其行为，如果容器资源配置影响了 Agent 可采用的策略空间，它也可能影响安全评测的结论。这是一个值得跟进的方向。
