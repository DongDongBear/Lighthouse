# 06. 推理时计算扩展：为什么多算一点会更准

## 核心概念

Test-time Compute（TTC）就是：

> 不改模型参数，只在推理时多花一些算力，换取更高质量答案。

## 最常见的三类方法

### 1) Best-of-N

同一个问题采样 N 次，选最好的一条。

优点：简单
缺点：依赖打分器质量

### 2) Majority Vote

N 条答案里，出现最多的那个作为最终答案。

优点：适合离散答案（例如实体名、数字）
缺点：对开放式长答案不友好

### 3) Generative Aggregation（KARL 的 Parallel Thinking）

先并行生成 N 条轨迹，再让模型读这 N 条结果，综合成最终答案。

优点：能融合互补信息，不只是“选一个”

KARL 论文里有个关键发现：在 PMBench 上，23.7% 情况下，聚合器给出的答案优于任意单条轨迹。

## Value-Guided Search（VGS）

再进阶一点：

- 每一步不是只走一个分支
- 并行生成多个候选动作
- 用 value model 给候选打分
- 选最有希望的继续

这相当于“推理过程中的搜索树剪枝”。

## TTC 的现实意义

- RL 训练提升“基础能力”
- TTC 提升“解题上限”
- 两者是叠加关系，不是替代关系

KARL 的结果说明：同一个模型通过增加并行轨迹，可以从接近 Sonnet 提升到匹配/超过 Opus。
