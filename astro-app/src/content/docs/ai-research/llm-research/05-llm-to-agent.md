# 05. 从 LLM 到 Agent：工具调用、多步推理、上下文管理

## LLM 和 Agent 的差别

- **LLM**：一次输入，一次输出
- **Agent**：可以循环“思考 → 调工具 → 观察结果 → 再行动”

Agent 的关键不在“会说话”，而在“会行动”。

## 一个典型 Agent 循环

1. 读当前问题和历史
2. 生成下一步动作（例如调用 vector search）
3. 执行动作，拿到结果
4. 判断是否继续搜索
5. 证据足够时给最终答案

这条完整过程就是一条 trajectory（轨迹）/rollout。

## Tool Use 是怎么接进去的

模型会输出结构化工具调用请求，例如：

```json
{"tool":"vector_search","query":"..."}
```

系统收到后执行工具，把结果作为新上下文喂回模型。模型再继续下一步。

## 为什么“上下文管理”是核心能力

多步 Agent 最容易死在上下文爆炸：

- 每搜一次就增加很多 token
- 搜 100 步后窗口可能爆掉

KARL 的做法：

- 触发阈值后让模型自己压缩历史
- 压缩步骤也纳入 RL 训练
- 让模型学会“为最终答对而压缩”

这比固定摘要器更强，因为它是任务驱动而不是通用摘要驱动。

## 读 KARL 时你该关注什么

看到这些词时，你现在可以对应上：

- rollout / trajectory：一次完整搜索过程
- turns / steps：轨迹中的动作步数
- compression：历史压缩，腾上下文
- tool call outputs masked：RL 只训模型生成部分，不训工具返回文本
