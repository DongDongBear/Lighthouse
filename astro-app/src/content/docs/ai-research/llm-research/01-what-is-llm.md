# 01. 大语言模型是什么

> 本文面向有编程基础但没有 ML 背景的开发者。目标是建立直觉，不深入数学。

---

## 最简单的理解

大语言模型（Large Language Model, LLM）本质上在做一件事：**给定前面的文字，预测下一个词。**

```
输入: "今天天气真"
模型预测: "好" (概率 0.45) / "不错" (概率 0.30) / "差" (概率 0.08) / ...
```

就是这样。ChatGPT、Claude、GPT-5 做的核心事情，就是反复预测下一个词。你输入一段话，它预测下一个词，把这个词加到输入里，再预测下一个词，如此循环，直到生成完整的回答。

看起来简单，但当你把这个机制做到极致——用万亿级别的文本训练、用千亿级别的参数——它就涌现出了推理、编程、翻译、创作等能力。

---

## Token：模型的"字母表"

模型不直接处理文字，而是处理 **Token**。Token 是模型的最小文字单元。

```
"Hello world" → ["Hello", " world"]        // 2 个 token
"你好世界"    → ["你", "好", "世", "界"]     // 4 个 token（中文通常一字一 token）
"unhappiness" → ["un", "happi", "ness"]    // 3 个 token（英文会拆分）
```

为什么不用字符？因为 token 在"太细"（字符）和"太粗"（完整单词）之间找到了平衡。模型的词汇表通常有 32K-150K 个 token。

**在 KARL 论文中**，你会看到 token 无处不在：
- "上下文窗口 1M token"——模型一次能处理的最大输入长度
- "rollout 的 token 长度 32,156"——Agent 一次搜索过程产生的 token 数
- "token 级别的掩码"——RL 训练时只计算模型自己生成的 token

---

## 模型参数：存储"知识"的地方

模型就是一大堆数字（参数），通常数十亿到数千亿个。

```
GPT-3:     1750 亿参数
GPT-4:     ~1.8 万亿参数（传闻）
GLM 4.5 Air: 未公开，但是 MoE 架构（稍后解释）
```

这些参数在预训练过程中被调整，使得模型能做出好的"下一个 token 预测"。你可以把参数理解为模型的"记忆"——它在海量文本中学到的所有模式、知识和能力，都编码在这些数字里。

---

## Transformer：核心架构

几乎所有现代 LLM 都基于 **Transformer** 架构（2017 年 Google 提出）。你不需要理解全部细节，但需要知道一个核心机制：**注意力（Attention）**。

### 注意力机制的直觉

想象你在读一段话：

> "小明昨天去了北京，他在那里参观了故宫。"

当你读到"那里"的时候，你的大脑会自动"关注"到前面的"北京"——这就是注意力。

Transformer 的注意力机制做的就是这件事：对于序列中的每个 token，计算它应该"关注"前面的哪些 token，以及关注多少。

```
"他"   → 关注 "小明"（因为"他"指代"小明"）
"那里" → 关注 "北京"（因为"那里"指代"北京"）
"故宫" → 关注 "参观" + "北京"（因为故宫在北京被参观）
```

这个机制让模型能理解长距离依赖关系——一个词和很远之前的词之间的关系。

### Self-Attention 的计算

Transformer 对序列中的每一对 token 计算"注意力分数"，决定它们之间的关联程度。对于长度为 n 的序列，需要计算 n×n 个分数。这就是为什么上下文窗口有长度限制——太长的序列，注意力计算的成本会爆炸（二次方增长）。

### 多层堆叠

一个 Transformer 模型由很多层（Layer）堆叠而成：

```
输入 token → Layer 1 → Layer 2 → ... → Layer 96 → 输出概率分布
```

每一层都有自己的注意力参数。浅层学到的是语法、词法等基础模式，深层学到的是语义、推理等高级能力。GPT-3 有 96 层，参数就分布在这些层中。

---

## MoE：混合专家模型

KARL 的基座模型 GLM 4.5 Air 使用了 **MoE（Mixture of Experts）** 架构。这个概念很重要，因为论文多次提到 MoE 训练的稳定性问题。

### 普通模型 vs MoE

**普通模型（Dense Model）**：每个 token 经过所有参数计算。如果模型有 1750 亿参数，每个 token 都要用到全部 1750 亿。

**MoE 模型**：把部分参数分成多个"专家"（Expert），每个 token 只激活其中几个。

```
Dense Model:
  token → [所有 1750 亿参数] → 输出

MoE Model:
  token → Router（路由器）→ 选择 Expert 2 和 Expert 7
                           → [Expert 2 的参数] + [Expert 7 的参数] → 输出
                           （其他 Expert 不参与计算）
```

好处：**总参数可以很大（知识容量大），但每个 token 的计算量小（推理快、成本低）**。

比如 DeepSeek-V3：总参数 6710 亿，但每个 token 只激活 370 亿。所以它有"大模型的知识量"和"小模型的计算成本"。

**在 KARL 论文中**，MoE 很重要是因为：GRPO（一种在线 RL 方法）在训练 MoE 时需要各种 trick（router replay、数据删除等）来保持稳定。KARL 的 OAPL 方法不需要这些 trick，这是它的核心优势之一。

---

## 上下文窗口：模型的"工作记忆"

模型一次能处理的最大 token 数叫做 **上下文窗口（Context Window）**。

```
GPT-3:    4K token
GPT-4:    128K token
GPT-5.4:  1M token
Claude 4.6: 200K token
```

你可以把上下文窗口想象成模型的"工作台"——它只能同时看到工作台上的文字。工作台越大，它能处理的信息越多。

**在 KARL 中**，上下文窗口管理是核心挑战。Agent 每做一次搜索，检索到的文档就会占用上下文空间。做到第 100 步时，累积的历史可能已经超出窗口。所以需要**压缩（Compression）**——让模型自己总结历史，腾出空间继续搜索。

---

## 推理过程：自回归生成

LLM 生成文本的过程叫做 **自回归生成（Autoregressive Generation）**：

```
Step 1: 输入 "请解释量子力学"
Step 2: 模型预测下一个 token → "量"
Step 3: 输入变为 "请解释量子力学 量"
Step 4: 模型预测下一个 token → "子"
Step 5: 输入变为 "请解释量子力学 量子"
...
Step N: 模型预测 → [结束标记]，停止生成
```

每一步都是独立的前向计算，所以生成速度和输出长度成正比。

**温度（Temperature）**：控制预测的随机性。

```
温度 = 0: 总是选概率最高的 token（确定性，适合事实性任务）
温度 = 0.7: 适度随机（适合对话）
温度 = 1.0+: 高随机性（适合创意写作）
```

**在 KARL 中**，推理时计算扩展（Test-time Compute）的核心就是：同一个问题，让模型多次生成（不同随机性产生不同轨迹），然后选最好的或者综合。

## 模型的能力边界

理解 LLM 的局限性同样重要：

**参数知识 vs 外部知识**

模型的知识来自训练数据，被编码在参数中。但：
- 训练数据有截止日期（模型不知道训练后发生的事）
- 企业私有数据没有被包含在训练中
- 模型可能"编造"看似合理但错误的信息（幻觉/Hallucination）

这正是 KARL 要解决的问题：模型需要**主动搜索外部数据**来回答问题，而不是仅依赖参数中的记忆。论文称这种需要外部知识的推理为 **Grounded Reasoning**。

**上下文学习 vs 参数学习**

模型可以通过两种方式获取新能力：
1. **上下文学习（In-Context Learning）**：把示例放在 prompt 里，模型现学现用，但不持久
2. **参数学习（训练/微调）**：修改模型参数，能力持久保存

KARL 的 RL 训练属于参数学习——永久性地改变了模型的搜索行为。
