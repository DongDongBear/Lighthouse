# 04. RAG 与向量搜索：为什么 LLM 需要“搜索”

## 为什么需要 RAG

模型参数里的知识是“静态记忆”，有三个问题：

1. 有截止日期（训练后新信息不知道）
2. 企业私有数据不在训练集中
3. 可能出现幻觉

RAG（Retrieval-Augmented Generation）就是让模型先查资料再回答。

## 最小流程

1. 用户问题 → 转成向量（embedding）
2. 去向量库检索最相似文档片段（chunks）
3. 把检索结果拼进 prompt
4. 模型基于证据生成答案

## 什么是 Embedding

Embedding 就是把文本映射成一个高维向量。

- 语义相近的文本，向量距离更近
- 检索时用余弦相似度或内积做近邻搜索

## Chunk 为什么重要

文档太长，不能整篇检索。要先切块（chunking）。

切太小：上下文断裂
切太大：检索不准、浪费窗口

KARL 里各数据集 chunk 方式不同，但原则一致：在可控预算里尽量保留可检索性。

## KARL 和普通 RAG 的区别

普通 RAG 常是“一次检索 + 一次回答”。

KARL 是 Agent 式：

- 多次检索
- 每次检索后更新判断
- 需要时继续搜
- 历史过长时压缩
- 最后再回答

所以 KARL 不是简单 RAG，而是“可迭代搜索推理系统”。
